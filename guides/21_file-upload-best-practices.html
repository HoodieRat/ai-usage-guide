File Upload Best Practices

Prepare documents, spreadsheets, transcripts, and media so the model ingests the right information without exposing restricted data. These practices are for ChatGPT’s web chat.

Do not upload restricted data unless it has been sanitized per Redaction & Sanitization. Files you attach may be referenced in the session and appear in exports or logs. See Memory & Privacy.

What you’ll learn

  • How to triage which files to upload, summarize, or keep offline.
  • Preparation steps that improve parsing accuracy and reduce token waste.
  • How to document uploads with manifests, retention tags, and follow-up reviews.

Limits & compatibility (read this)

  • Plan limits change. Check the chat uploader tooltip for current size/type limits and files-per-message caps.
  • Commonly supported types: .txt, .md, .pdf, .docx, .xlsx/.csv, .pptx, .png/.jpg. Complex PDFs or scans may need OCR.
  • Right-size batches: If a file is long or image-heavy, split into thematic parts (≈10–20 pages) and attach with summaries.
  • Prefer text: When possible, convert PDFs and slides to clean text or Markdown to improve recall and reduce latency.

If a thread slows or stalls after many uploads, start a new chat and paste a recap + file manifest (see Sessions & Context).

Prerequisites

  • Project folder manifest (see Project Folders Setup).
  • Redaction tooling or templates for sensitive data (guide).
  • Awareness of current chat uploader limits and allowed file types.

Planner → Actor → Reviewer walkthrough

Planner prompt

“We need to upload discovery call notes, a pricing spreadsheet, and a draft proposal. Decide what to redact, how to label each file, and the order to upload so the assistant can answer pricing questions.”

Actor response

The assistant produces a manifest with classification, redaction instructions, summaries, and the upload sequence.

Reviewer check

The reviewer verifies redactions, confirms summaries retain intent, and asks the assistant to draft a confirmation note listing file IDs and next actions.

Step-by-step

  1. Inventory files. List every potential asset with owner, source, and data class (public/internal/confidential/restricted).
  2. Prep content. Convert to UTF-8 text where possible, flatten tracked changes, and remove hidden metadata/comments.
  3. Summarize & label. Write 3–5 bullet abstracts and assign IDs (F1, F2) that match your prompts and handoffs.
  4. Upload in batches. Attach related files together, paste their summaries, and ask the assistant to confirm receipt before moving on.
  5. Log & review. Update the manifest with timestamps, chat links, and follow-up redactions or deletions.
Format-specific tips
  • PDFs: Prefer text-native PDFs. For scans, run OCR and spot-check headings/tables. Consider exporting to .txt/.md.
  • Docs (.docx): Accept/reject tracked changes; remove comments; export to .md for long policy docs.
  • Sheets (.xlsx/.csv): Keep a “ReadMe” sheet with column definitions; export analytical tables to .csv; avoid hidden sheets.
  • Slides (.pptx): Export speaker notes; consider a single-page brief with key bullets + slide thumbnails only when necessary.
  • Images: Provide a caption/legend describing what to read from the image (e.g., “read axis labels; ignore legend 2”).

Output contract

When you ask the assistant for upload guidance, require the reply to include:

  • Purpose: The question or workflow the file supports.
  • Data class: Whether redaction or masking is needed.
  • Summary: A concise abstract to paste when uploading.
  • Retention plan: Timeline to delete or archive the file.
  • Confirmation step: Exact wording to verify the assistant processed the asset.

We are uploading {{count}} files. For each file I will provide:

- Filename

Redaction quick checklist

[ ] Replace names with role placeholders
[ ] Remove contract numbers, account IDs, and email addresses
[ ] Strip comments, track changes, and hidden sheets
[ ] Convert PDFs to text and verify OCR accuracy
[ ] Re-run spell check after redaction to catch placeholder typos

Upload confirmation snippet (paste after each batch)

Files attached: F1, F2.
Summaries pasted above. Confirm you can read them and list:
1) 3 key takeaways per file
2) Risks or policy conflicts
3) Any missing context to answer {{goal}}

Logging

Store the manifest in your project folder. Capture upload time, chat link, redaction owner, and deletion confirmation. For regulated data, export the assistant’s confirmation messages and archive them with the manifest.

Metrics that matter

  • Prep-to-upload time: Minutes from selecting a file to completing a sanitized upload (target ≤ 15 min for standard docs).
  • Redaction defect rate: Incidents where sensitive data slipped through post-upload review.
  • Confirmation compliance: Percentage of uploads with logged assistant acknowledgements.

Tips & pitfalls

  • Tip: Keep a running manifest as you prep files so owners and reviewers can see redaction status at a glance.
  • Tip: Paste a short summary right after each upload—confirmation requests work better when the assistant sees fresh context.
  • Pitfall: Ignoring size caps until the upload fails. Split or compress large assets first and record the workaround in the manifest.
  • Pitfall: Forgetting hidden metadata. Strip comments, revision history, and hidden sheets before ChatGPT ever sees the file.

Recap

Thoughtful uploads keep tokens low, privacy intact, and collaboration efficient. Plan the batch, sanitize each file, label and summarize it, confirm ingestion, and log every step. For very long sources, pair this with Chunking & Abstracts and retrieval patterns in Context from Files (RAG Basics).

Acceptance checklist

  • ✅ Manifest template created and linked from the project README.
  • ✅ Redaction checklist reviewed with the team.
  • ✅ Upload confirmation script saved as a reusable snippet.
  • ✅ Metrics sheet tracking prep time, defects, and confirmations.
  • ✅ Retention schedule defined for uploaded assets.
