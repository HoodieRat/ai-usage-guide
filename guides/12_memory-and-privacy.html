Memory & Privacy

Use ChatGPT’s Memory features intentionally. Decide what may persist across chats, how to minimize sensitive data, and how to purge quickly—without losing productivity on long-running projects.

Retention reminder: Company policy caps AI memory at 30 days for any customer-identifiable data. Record every exception with an owner, purge date, and privacy approval.

What you’ll learn

  • What Memory is good for (reusable preferences) and what it is not (private data vault).
  • How to combine Custom Instructions, Context Transfer, and Memory safely.
  • How to classify, redact, log, and purge memory entries for audits.

Quick wins

Prefer preferences, not facts. Save style/format defaults in Memory (e.g., “prefer tables, keep answers under 150 words”), not client secrets.

Use Temporary Chat for sensitive tasks. For blank-slate work that shouldn’t inherit Memory, start a temporary chat and attach only needed files.

Restate the rules. Start important runs with a one-line privacy primer (copy-paste below) so the model doesn’t drift.

How Memory fits with sessions

Feature What it does Use it for Avoid storing
Memory Lets ChatGPT recall certain details across chats. Voice/tone, preferred formats, recurring non-sensitive preferences. Credentials, customer identifiers, health/financial data.
Session context Short-term chat history used for the current thread only. Task steps, drafts, in-progress reasoning. Long-term facts you won’t need after this task.
Context Transfer Manual recap you paste to start a fresh chat cleanly. Project digests (key facts, links, next steps). Any sensitive data not required for the next step.

If a thread slows or “locks up,” start a new chat and use the Context Transfer block.

Step-by-step

  1. Classify first. Tag incoming info as public/internal/confidential/restricted (see policy). If in doubt, treat as restricted.
  2. Decide storage. For each item, pick: Allowed in Memory, Session only, or Blocked. When blocked, rely on attachments + session context.
  3. Prime the run. Paste the guardrail primer (below) at the top of the chat if stakes are high.
  4. Save only what’s safe. If you add a Memory note, include its classification and a purge date in your log.
  5. Review weekly. Open the Memory list, remove anything stale, and record the purge in your privacy log.

Privacy guardrail primer (copy-paste)

Operate in privacy-safe mode:
- Do not store confidential or restricted information in Memory.
- If uncertain, ask for classification before saving anything.
- Prefer storing reusable preferences (tone, formatting) only.
If a store request is risky, refuse and explain the conflict.

Redaction & sanitization

Before saving notes—or even before pasting context—remove identifiers and sensitive fragments. Pair with the dedicated guide: Redaction & Sanitization.

Redaction checklist:
- Replace account IDs with aliases (e.g., ACME-123 → ACME-A1).
- Remove payment, credential, or medical details entirely.
- Limit transcripts to pertinent, non-identifying excerpts.
- Note which sanitized file is the “source of truth”.

Operational playbook

  1. Classify inputs. Use your matrix: public / internal / confidential / restricted.
  2. Decide storage scope. Record “Memory / Session / Block” per class with rationale.
  3. Prime the assistant. Paste the primer before attaching files for sensitive tasks.
  4. Log exceptions. If restricted data must persist briefly, log owner, reason, and purge-by date.
  5. Review & purge. Weekly: export the Memory list, mark Keep/Edit/Delete, purge, and log proof.

Output contract for “store this in Memory”

When you ask the model whether to save a detail, require this structure in its reply:

  • classification (policy tier + citation)
  • allowed (true/false with reason)
  • retention_days (number; 0 if not allowed)
  • deletion_phrase (exact phrase to remove this memory later)
  • reviewer (role/name approving the entry)
JSON example 
{
  "classification": "internal",
  "allowed": true,
  "retention_days": 30,
  "deletion_phrase": "Forget my preference about weekly summary format.",
  "reviewer": "Privacy steward"
}

Copy & paste assets

Emergency purge (paste into the chat)

List everything you currently remember about me and this project.
Identify any items that could be confidential or restricted.
Then: confirm you have forgotten those items now.

Weekly purge runbook

# Weekly Memory Review
1) Open the Memory list and export/copy entries.
2) Tag each as Keep, Edit, or Delete in the privacy log.
3) Issue deletion phrases in-chat; confirm removal.
4) Attach the confirmation to the log with date/reviewer.
5) Escalate any item that cannot be removed.

Logging & audit

Track every Memory write or purge with timestamp, author, data class, purge date, and reviewer initials. Keep exports or screenshots attached to the log so reviewers can rehydrate context quickly.

Metrics that matter

  • Exception rate: % of chats requesting restricted storage (target < 5%).
  • Time-to-purge: Average hours from deletion request to confirmed removal.
  • Repeat incidents: Recurrent policy violations per quarter.

Tips & pitfalls

  • Tip: Before sensitive runs, ask “What do you currently remember about me?” and clear anything off-mission.
  • Tip: Schedule a weekly Memory review on your team calendar so purges never slip.
  • Pitfall: Letting restricted data persist “just this once.” Run the emergency purge immediately and log the incident.
  • Pitfall: Skipping proof of deletion. Capture screenshots or exports after each purge in the privacy log.

Recap

Save preferences, not secrets. Classify before storing, prime guardrails, log changes, and purge routinely. For long projects, avoid Memory bloat—use Context Transfer and Handoff Between Chats instead of hoarding details in Memory.

Acceptance checklist

  • ✅ Data classification matrix accessible and used.
  • ✅ Guardrail primer pasted for sensitive runs.
  • ✅ Logging and weekly purge cadence in place with an owner.
  • ✅ Memory “store” decisions include class, retention, deletion phrase, reviewer.
  • ✅ Last audit exported and linked in the privacy log.
