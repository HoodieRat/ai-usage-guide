Version Control for Prompts

Treat prompts like code so history, approvals, and rollback paths stay intact.

Pair with: Prompt Handovers for smooth transfers and Knowledge Bases for documentation.

What you’ll learn

  • Organizing prompts into repositories with branching strategies.
  • Designing commit messages that capture context.
  • Running reviews and approvals before shipping prompt updates.
  • Tagging releases so support teams know what changed and when.

Prerequisites

  • Platform with history and permissions (Git or equivalent).
  • Repo structure agreed by product/area/use case.
  • Review checklist and evaluation harness.

Step-by-step

  1. Pick source of truth. Git, Notion export, or Airtable with versioning.
  2. Design structure. Folders by product/persona/workflow + README.
  3. Branching rules. Experiments on feature/; production on main.
  4. Metadata. Front matter: owner, status, last review, dependencies.
  5. Reviews. Pull requests with checklists and automated evals.
  6. Tags & changelog. Semantic or calendar versions with impacts.

Output contract

Your prompt repo should include:

  • README: scope, structure, labels, and branching.
  • Templates: prompt files with front matter and eval harness.
  • Protection rules: code owners, required reviews, CI checks.
  • CHANGELOG: human-readable history tied to tags.

Front matter (YAML)

---
owner: team@company.com
status: draft | needs-review | approved
last_review: 2025-10-30
dependencies: [policy/brand.md, dataset/customer_faq.csv]
---

Commit message convention

feat(prompt): add escalations to support triage
refactor: split planner/actor prompts for clarity
chore: update last_review dates

CHANGELOG

## 2025-10-30
- feat: Introduced escalation language to /support/triage_v3.md
- fix: Corrected entity extraction rules in /sales/demo_v2.md

Logging

Store pull request links, reviewers, evaluation results, and tag notes so support can audit what shipped.

Metrics that matter

  • Lead time to change: open → merge.
  • Review coverage: % of changes merged with approvals and evals.
  • Rollback frequency: tags reverted within 7 days.

Troubleshooting

Prompt drift across tools.
Automate exports from product UIs into the repo nightly.
Blocked reviews.
Define backup reviewers and SLA; allow emergency hotfix tags with postmortems.
Unclear history.
Require changelog entries in the PR template.

Recap

Treat prompts like code. With structure, reviews, and tags, you’ll know exactly what shipped and how to roll it back.

Acceptance checklist

  • ✅ Repo structure published with README and templates.
  • ✅ Branch protection and reviews enforced.
  • ✅ Evaluations run before merges.
  • ✅ Releases tagged and changelog updated.
