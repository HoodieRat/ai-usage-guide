Custom Instructions

Set durable defaults so every new chat starts with your voice, priorities, and guardrails. This play shows how to author, review, and maintain Custom Instructions your team can trust—without overfitting or getting “sticky.”

What you’ll learn

  • Collect policy inputs and brand cues that make instructions durable.
  • Structure the “About you” and “How ChatGPT should respond” blocks for clarity and reuse.
  • Pair instructions with a reviewer workflow and light versioning.
  • Avoid common pitfalls (verbosity, brittle rules, conflicts with Memory).

Quick wins

Keep it short. Aim for ≤ 200–250 words total across both fields. Short beats perfect.

Name the audience. Saying who the output is for instantly improves tone and detail level.

Pin the format. Add 1–2 “always use” formats (bullets, tables, JSON) you rely on most.

Step-by-step

  1. Gather source material. Voice/brand notes, forbidden data, escalation paths, and top recurring tasks.
  2. Draft both fields. Use bullets and constraints; avoid long prose and vague adjectives.
  3. Reviewer pass. Have someone else check for risk coverage, brevity, and clarity.
  4. Version & store. Save a copy (Markdown or JSON) with owner, version, and next review date.
  5. Test on 3 real tasks. If you keep retyping the same follow-up, your instructions need a tweak.
  6. Revisit monthly. Remove cruft; keep what demonstrably reduces back-and-forth.

Templates (copy-paste)

Personal — concise default

About you (who you are & what you want):
- Role: <designer / developer / manager / student>
- Audience: <self / stakeholder / non-technical / expert>
- Work style: brief, practical, examples-first
- Preferences: plain language, no hype, show steps only when asked
- Don’t: invent sources, over-explain, or change file names

How ChatGPT should respond (rules & formats):
- First, restate my goal in one line and list missing info (max 1 question).
- Prefer bullets or short tables; keep outputs ≤ 200 words unless I say longer.
- When facts matter, mark uncertainties and suggest a quick verification step.
- Use relative links for this project and maintain the requested format strictly.

Team — brand & governance

About you:
- We are <Team/Org> building <product/guide> for <audience>.
- Voice: clear, calm, friendly; avoid hype and sarcasm.
- Compliance: do not include personal data; no speculative claims.

How ChatGPT should respond:
- Start with a 1-line objective check. If unclear, ask one clarifier, then proceed.
- Structure: bullets or Markdown tables unless JSON is requested.
- Limits: title ≤ 80 chars; summaries ≤ 150 words; keep filenames unchanged.
- Verification: flag risky claims; propose a source or note “needs verification”.
- Escalation: if out-of-scope or missing inputs, say so and list exactly what is needed.

Ops — format-heavy (for structured outputs)

About you:
- Role: operations analyst producing checklists and CSV-ready tables.
- Audience: internal teammates who will copy-paste into sheets.

How ChatGPT should respond:
- Return ONLY a Markdown table with columns: Item | Owner | Due | Status.
- If info is missing, leave blank; do not invent values.
- Add a final line “Notes:” with 1 sentence, or leave it empty.

Reviewer checklist (PASS / REVISION)

  • PASS if total words ≤ 250 and both fields use bullets.
  • PASS if audience, tone, and top formats are explicit.
  • REVISION if it duplicates things better handled by Context Transfer.
  • REVISION if it conflicts with Memory & Privacy practices.
  • REVISION if it hard-codes project-specific facts that will change.
  • REVISION if it buries the important rules under paragraphs of prose.

Versioning & change log (lightweight)

id: custom-instructions.v1
owner: <name>
version: 1.1.0
last_review: 2025-10-30
notes: Shortened; added verification rule; clarified formats.
changelog:
  - change_id: CI-2025-10-30-a
    summary: Tone tightened; added out-of-scope behavior
    approver: <name>

Store alongside your prompt cards in Prompt Library. Link from tickets/wikis so teammates use the current version.

Custom Instructions vs. Memory vs. Context Transfer

  • Custom Instructions = standing defaults for every new chat (tone, audience, formats).
  • Memory = facts ChatGPT may carry across chats; use sparingly to avoid “sticky” behavior. See Memory & Privacy.
  • Context Transfer = paste a short recap + links when you start a fresh chat or hand off. See Sessions & Context and Handoff Between Chats.

Tips & pitfalls

  • Tip: Reuse bullet templates for recurring roles so teammates can adapt them quickly.
  • Tip: Add “If you need more info, ask one clarifying question” to reduce back-and-forth.
  • Pitfall: Letting instructions sprawl past 250 words. Trim or split into project-specific context packs.
  • Pitfall: Conflicts with Memory. Prefer Temporary Chat or restate which rule wins when they disagree.

Keep going

  • Prompt Basics — five-line requests that always land.
  • Response Contracts — enforce formats and error paths.
  • Structured Output — JSON/tables that validate cleanly.
  • Memory & Privacy — when to save vs. forget.

Acceptance checklist

  • ✅ Both fields are under ~250 words and mostly bullets.
  • ✅ Audience named, tone set, 1–2 default formats specified.
  • ✅ Verification / escalation behavior included (one line each).
  • ✅ Saved with version, owner, and next review date.
  • ✅ Tested on 3 real tasks without constant follow-ups.
